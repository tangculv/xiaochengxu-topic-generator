# 小程序生产线系统 PRD

> 批量生产可变现的微信小程序，用数据筛选赢家，让试错成本足够低。

---

## 1. 问题

一个人做小程序，最大的浪费不是开发慢，而是：

- **做错了方向**——选题靠直觉，没有市场信号验证
- **每次从零开始**——支付、分享、埋点反复造轮子
- **AI 生成的页面不能用**——排版乱、导航崩、设计丑，返工量比写代码还大
- **上线后看不见数据**——无法判断该加注还是该砍
- **砍不下手**——没有客观标准，资源耗在僵尸产品上

这些问题的共同根源：**没有一套可复用的生产机制。** 本 PRD 要解决的就是这个。

---

## 2. 系统全景

### 2.1 一句话定位

> 面向"可变现、可分享、可追踪"的微信小程序生产线。开发规范是核心资产，小程序是产出。

### 2.2 并行运行模型

任意时刻，多个小程序处于不同阶段，三条带同时运转：

```
     选题带              开发带              运营带
  ┌────────┐         ┌────────┐         ┌────────┐
  │ 评估候选 │         │ Codex  │         │ 监控数据 │
  │ 生成PRD  │  ──→   │ 开发中  │  ──→   │ 推广执行 │
  │ 立项决策 │         │ 质检中  │         │ 淘汰/毕业│
  └────────┘         └────────┘         └────────┘
  不限数量             同时 1-2 个         最多 5 个
```

三条带独立运转，互不阻塞。操作者每天的工作节奏：
1. 先看运营带——日报有没有需要处理的建议
2. 再推开发带——Codex 当前进度，需要预览/确认什么
3. 有空余时间——选题带补充候选

### 2.3 工具链

| 角色 | 工具 | 职责 |
|------|------|------|
| 文档 | **Claude Code** | 选题评估、PRD 生成、推广方案、运营分析 |
| 开发 | **Codex** | 代码实现、构建、自检 |
| 预览 | **微信开发者工具** | 操作者查看开发结果 |
| 运营 | **Obsidian** | 查看日报、看板、处理建议 |

### 2.4 硬约束

**后端**：所有小程序统一微信云开发。不支持自建服务端。

**产品形态**——只做这几种页面：

| 类型 | 例子 |
|------|------|
| `form-to-result` | 测评、食谱生成、计算器 |
| `list-detail` | 内容浏览、记录查看 |
| `dashboard` | 数据概览 |
| `record-history` | 打卡、记账、日记 |
| `chat-lite` | AI 问答 |
| `paywall-result-share` | 付费解锁内容 |

**不做**：多角色后台、审批流、音视频、地图重交互、重交易履约、线下服务调度。

## 3. 开发规范（核心资产）

> 规范向上约束 PRD 怎么写，向下约束 Codex 怎么开发。规范是这条生产线最值钱的东西。

### 3.1 规范必须解决的问题

| 问题 | 规范必须做到 |
|------|------------|
| AI 生成的页面丑、乱 | 锁死视觉层：Codex 不做设计决策，只用预定义的组件和布局拼装页面 |
| 底部导航不稳定 | TabBar 使用原生配置，模板级锁定，Codex 不能自己实现导航 |
| 页面跳转出 bug | 统一路由封装，所有跳转走同一个入口，路径必须注册 |
| 支付对接反复踩坑 | 支付模块封装为黑盒，Codex 只调接口，不碰实现 |
| 分享裂变不稳定 | 分享模块封装为黑盒，分享卡片只有预设样式可选 |
| 每个小程序埋点不一致 | 埋点 SDK 自动采集生命周期事件，业务事件统一 API |
| 不同小程序的代码风格不一致 | 框架、样式方案、目录结构、命名规范全部统一 |

### 3.2 规范的构成

规范不是一份文档，而是四样东西配合工作：

```
┌─────────────────────────────────────────────────┐
│                    开发规范                        │
│                                                   │
│  ① 模板项目        拿来就能跑的 Taro 项目            │
│     含 auth / payment / growth / analytics 模块    │
│     含组件库 / 布局模板 / design tokens             │
│     含云函数模板（crud / ai-generate）              │
│                                                   │
│  ② PRD-RULES.md    告诉 Claude Code 怎么写 PRD     │
│     PRD 的标准格式、必填项、引用范围                   │
│     确保 PRD 中的每一项都能对应到模板项目中的能力        │
│                                                   │
│  ③ CODEX-RULES.md  告诉 Codex 怎么写代码            │
│     必须使用哪些组件、禁止哪些行为                     │
│     每个模块完成后的自检清单                          │
│                                                   │
│  ④ CHECK-RULES     check 脚本的检查规则              │
│     blocker / warning / manual 三级                │
│     所有自动化可验证的约束都在这里                      │
│                                                   │
│  四者保持一致：PRD 中能写的 = Codex 能做的            │
│              = 模板中有的 = check 能验的              │
└─────────────────────────────────────────────────┘
```

> **关键原则**：四者的范围必须一致。如果 PRD 中写了一种页面类型，模板项目中必须有对应的布局模板，CODEX-RULES 中必须有对应的开发指引，check 脚本中必须有对应的验证规则。四者不一致时，以模板项目的实际能力为准。

### 3.3 规范在 M0 阶段确立

规范不是先写好再开发，而是**在做第一个小程序的过程中沉淀出来的**：

```
做第一个小程序
  ├── 发现需要哪些组件 → 沉淀为组件库
  ├── 发现什么布局好看 → 沉淀为布局模板
  ├── 发现什么 token 值合适 → 沉淀为 design tokens
  ├── 发现 Codex 容易犯什么错 → 沉淀为 CODEX-RULES
  ├── 发现 PRD 需要写到什么程度 → 沉淀为 PRD-RULES
  └── 发现哪些检查能自动化 → 沉淀为 check 脚本
```

所以 M0 的产出不只是"第一个小程序"，更重要的是**经过实战验证的规范体系**。

### 3.4 规范的硬性要求

以下是规范必须满足的要求（不是规范的具体内容——具体内容在 M0 中确定）：

**架构**：
- 框架统一（Taro 3.x React）
- 样式方案统一（禁止内联样式、禁止全局样式污染）
- 一项目一云环境，云函数从模板实例化
- 环境变量统一注入，不在代码中硬编码

**视觉**：
- 有 design token 系统（颜色、字体、间距、圆角、阴影）
- 每个小程序仅通过配置切换主色，其余 token 共享
- Codex 引用 token 名称，禁止使用原始值
- 有预定义组件库，Codex 禁止自己写 UI 原语
- 每种支持的页面类型有对应的布局模板

**导航**：
- 底部 TabBar 使用 Taro 原生配置，2-4 个 Tab
- 页面跳转统一封装，禁止直接调用 Taro API
- 所有跳转路径必须在配置中注册

**模块**：
- 支付：统一入口，付费墙统一组件，回调统一云函数
- 分享：统一组件，只有预设样式可选，统一追踪
- 埋点：自动采集生命周期事件，业务事件统一 API，统一上报

## 4. 工作流程

### 4.1 选题带：想法 → PRD

**输入**：一句话选题 + 大概功能描述

**步骤**：

**① 市场信号验证**（手动，来自推广 SOP）

```
✅ 微信搜一搜有相关小程序且有使用量    +2
✅ 小红书相关笔记点赞总量 > 5000      +1
✅ 竞品体验差/功能缺失                +1
✅ 目标用户愿意为此付费               +1
❌ 完全没有竞品                      -1
总分 ≥ 3 → 进入下一步
```

**② AI 评估**：按 8 大标准评分，≥70 分进入候选池

**③ 立项闸门**：8 项全过才允许生成 PRD

| # | 检查项 |
|---|--------|
| 1 | 有明确目标用户 |
| 2 | 有明确核心动作 |
| 3 | 首次使用 3 步内完成核心价值 |
| 4 | 有明确付费点 |
| 5 | 有明确分享触发点 |
| 6 | 至少 1 个可定义的核心完成事件 |
| 7 | 页面形态在支持列表内 |
| 8 | 可完全基于微信云开发实现 |

**④ Claude Code 生成 PRD**：基于 `PRD-RULES.md` 生成标准化 PRD

PRD 标准格式（由 PRD-RULES.md 约束）：

```
# [名称] PRD

## 产品定位（一句话）
## 目标用户（画像 + 场景）
## 页面结构
  - 页面列表（每页标注类型，必须在支持列表内）
  - TabBar 配置
## 用户流
  - 核心路径
  - 付费路径
  - 分享路径
## 各页面详细描述
  （用途、输入输出、应使用的布局模板、包含的组件）
## 数据模型
## 云函数 API（从预设模板选择）
## 埋点事件
## 付费方案（模式 + 价格 + 触发点 + 免费额度）
## 分享方案（样式 + 触发页 + 邀请码）
## 配置项（factory.config.yaml 的关键字段）
```

> PRD 中的每一项都必须能对应到模板项目中的实际能力。PRD 中出现模板不支持的东西，说明要么 PRD 写错了，要么这个选题不适合这条生产线。

**⑤ 操作者审核 PRD → 确认 → 交给 Codex**

---

### 4.2 开发带：PRD → 可预览的小程序

**输入**：审核通过的 PRD + 模板项目 + CODEX-RULES.md

**Codex 的工作方式**：

```
读取 PRD + CODEX-RULES.md
  ↓
不确定的地方先问操作者
  ↓
拆解开发任务，输出任务列表 + 依赖关系
  ↓
操作者确认
  ↓
按模块开发，每完成一个模块：
  ├── 自检（编译通过、符合规范、功能正确）
  └── 操作者在开发者工具中预览
  ↓
全部完成 → 运行 check → 修复问题
  ↓
满足上线标准 → deploy
```

**开发顺序**：

| 阶段 | 内容 | 自检 | 操作者动作 |
|------|------|------|-----------|
| 基座 | 克隆模板、填配置、配 TabBar 和主题色 | 启动成功 | 预览：看到空壳应用 |
| 页面 | 按 PRD 逐页开发，使用组件库+布局模板 | 编译通过、组件来源合规 | 每页预览 |
| 云函数 | 从模板实例化，配置集合名和字段 | 本地调用测试 | — |
| 接入 | 支付/分享/埋点按规范接入 | 各流程跑通 | 测试支付/分享 |
| 联调 | 前后端联调，完整路径走通 | 核心路径+付费路径+分享路径 | 端到端预览 |

**Codex 禁止的行为**（CODEX-RULES.md 核心内容）：
- 禁止自己写 UI 原语（button、card、list-item 等必须用组件库）
- 禁止内联样式
- 禁止自己实现底部导航
- 禁止直接调用 Taro 跳转 API
- 禁止在代码中硬编码环境变量或密钥
- 禁止自己写支付/分享/埋点逻辑（必须用模块封装的接口）

### 4.3 上线标准

一个小程序从开发带进入运营带，必须满足：

| # | 条件 | 验证方式 |
|---|------|---------|
| 1 | check 脚本无 blocker | 自动 |
| 2 | check 脚本 warning 已人工确认 | 人工 |
| 3 | 核心用户路径可跑通 | 人工走一遍 |
| 4 | 至少 1 个核心事件可触发并上报 | 查云数据库 |
| 5 | 支付链路已联调或明确 mock 验证 | 人工 |
| 6 | 隐私协议/授权流程已确认 | 人工 |
| 7 | 在 2-3 种机型上显示正常 | 人工预览 |

**check 脚本三级分类**：

| 级别 | 含义 | 处理 |
|------|------|------|
| `blocker` | 不过不能上线 | 必须修复 |
| `warning` | 可上线，但需确认 | 人工确认可接受 |
| `manual` | 脚本无法检测 | 人工逐项确认 |

blocker 包括：编译失败、硬编码密钥、使用了非组件库 UI、TabBar 非原生配置、跳转路径未注册、cloud_env_id 未配、必需云函数缺失、环境变量占位符未替换、包大小超限。

### 4.4 运营带：数据 → 决策

**自动运行**：埋点 SDK 采集 → 云函数每日聚合 → 生成 Markdown 日报 → Obsidian 查看

**核心指标**：

| 指标 | 及格线 | 数据源 |
|------|--------|--------|
| 核心功能完成率 | > 20% | 核心完成事件 / DAU |
| 次日回访率 | > 10% | Day2 回访 / Day1 新增 |
| 自然分享率 | > 2% | 分享事件 / DAU |

> 3 项中 ≥ 2 项达标 = 验证通过。阈值在前 5 个小程序中校准。

**检查点与建议**：

| 时间 | 条件 | 建议 |
|------|------|------|
| Day 7 | DAU < 3 连续 3 天 | ⚠️ 检查搜一搜排名 |
| Day 30 | 完成率 < 10% | 🔴 建议淘汰 |
| Day 30 | ≥ 2 项达标 | 🟢 进入验证期 |
| Day 60 | 付费率 < 1% 且 DAU < 20 | 🔴 建议淘汰 |
| Day 90 | ≥ 2 项达标 + 月收入 > 0 | 🟢 建议毕业 |
| 任意 | 连续 7 天 DAU = 0 | 🔴 建议下架 |

**建议处理**：每条建议必须记录 `accept / reject / defer`。reject 和 defer 必填原因，defer 必填复查日期。

**在库看板**（每日自动生成）：

```markdown
# 生产线看板 — 2026-04-15

## 运营带 [3/5]
| 小程序 | 阶段 | DAU | 完成率 | 留存 | 分享率 | 月收入 | 天数 |
|--------|------|-----|--------|------|--------|--------|------|
| 智能食谱 | 验证 | 45 | 28% | 12% | 3.1% | ¥380 | D62 |
| 记工本 | 观察 | 12 | 15% | 8% | 1.2% | ¥0 | D25 |
| 功德箱 | 预警 | 3 | 4% | 2% | 0.5% | ¥0 | D31 |

## 待处理
- [建议淘汰] 功德箱：Day30 完成率 4% → accept / reject / defer?

## 本周动作
- 淘汰：功德箱（若 accept）→ 释放 1 个槽位
- 启动：候选池最高分 91，可评估立项
- 关注：记工本接近 Day30 检查点

## 开发带 [1]
- 作文批改：Codex 开发中，页面完成 3/5

## 选题带
- 候选池：8 个（最高 91 分）
```

### 4.5 推广：每个小程序的独立方案

每个小程序上线时，由 Claude Code 生成独立推广方案：

**推广方案标准格式**：

```
# [小程序名] 推广方案

## 目标用户画像（从 PRD 继承）
## 渠道选择
  - 主推渠道（2-3 个，说明为什么选这些）
  - 各渠道预期效果
## 搜一搜优化
  - 核心关键词（3-5 个）
  - 名称/描述优化建议
## 内容策略
  - 各渠道的内容方向
  - 发布节奏
## 物料清单
  - 分享卡片文案（2-3 版）
  - 小红书笔记模板（1-2 篇）
  - 知乎回答模板（1 篇）
  - 微信群/朋友圈话术（1-2 版）
## 预算（遵循推广 SOP）
  - 第一个月：0 元
  - 第二个月：200 元（选 1 个渠道验证）
  - 第三个月：≤ 300 元（放大有效渠道）
## 效果判断标准
  - 什么数据说明渠道有效 → 加注
  - 什么数据说明渠道无效 → 停止
```

**推广效果反馈**：推广数据（渠道来源、获客成本）纳入运营带日报，用于：
- 判断哪个渠道有效，调整预算分配
- 同类型小程序的推广经验复用（通过知识库）

## 5. 知识积累

> 批量做小程序最大的复利是"做多了越来越准"。知识库是这个复利的载体。

### 5.1 积累什么

| 类型 | 来源 | 存入时机 |
|------|------|---------|
| 选题判断 | 评估分 vs 实际表现 | 小程序毕业或淘汰时 |
| 死因分析 | 被淘汰小程序的失败原因 | 淘汰时 |
| 开发踩坑 | Codex 反复犯的错、规范遗漏 | 每个小程序开发完成时 |
| 推广经验 | 哪种类型的小程序在哪个渠道有效 | 推广周期结束时 |
| 组件缺口 | 开发中发现需要但组件库没有的 UI | 开发过程中 |

### 5.2 怎么用

| 积累 | 反馈给 |
|------|--------|
| 选题判断 | 调整 8 大标准的权重 |
| 死因分析 | 更新立项闸门的检查项 |
| 开发踩坑 | 更新 CODEX-RULES.md |
| 推广经验 | 更新推广方案生成模板 |
| 组件缺口 | 扩展组件库 |

### 5.3 节奏

- **每个小程序完成开发**：记录开发踩坑 + 组件缺口
- **每个小程序淘汰/毕业**：记录选题判断 + 死因/成因 + 推广经验
- **每 3 个小程序做完**：回顾知识库，看是否需要调整选题标准权重或更新规范

### 5.4 存储

MVP 阶段用 Markdown 文件，每个小程序的 NOTES.md 记录过程，淘汰/毕业时归档到知识库：

```yaml
# 淘汰归档模板
title: "情绪功德箱"
score: 82
```

## 6. 毕业与淘汰

```
运营带：上线 → 90 天验证
                        │
           ┌────────────┤
           ▼            ▼
      淘汰              毕业
      ├─ 释放槽位         ├─ 独立仓库
      ├─ 归档死因         ├─ 独立运营策略
      ├─ 更新知识库       ├─ 精细化打磨
      └─ 推广停止         └─ 推广持续
```

**毕业标准**：
- 连续 30 天 ≥ 2 项核心指标达标
- 月收入 > ¥500
- 用户增长趋势为正

毕业后脱离运营带，释放槽位。

## 7. 里程碑

> 只定义内容和验收，不预估时间。按依赖关系串行推进。

### M0：规范体系 + 模板项目 + 第一个小程序

**核心目标**：通过做第一个小程序，建立经过实战验证的开发规范体系。

## 8. 风险

| 风险 | 应对 |
|------|------|
| Codex 不遵守 CODEX-RULES | 规范中加负面示例 + check 脚本强制校验 |
| 组件库覆盖不够（遇到新 UI 需求） | 先扩展组件库，再做具体小程序。记入知识库 |
| 设计系统在某些机型上不一致 | M0 阶段在 3 种主流机型上验证 |
| 云函数安全/幂等问题 | 从审计过的模板实例化，不全自动生成 |
| 微信审核政策变更 | 每个小程序独立主体/AppID |
| 三条带并行超出精力 | 运营带自动化程度高（自动日报+自动提醒），选题带按需推进 |

**需要验证的假设**：

| 假设 | 验证方式 |
|------|---------|
| 设计系统能让 Codex 生成的页面"不丑" | M0 第一个小程序验证 |
| CODEX-RULES 能有效约束 Codex 行为 | M0 统计规范违反率 |
| PRD-RULES 生成的 PRD 可直接交给 Codex | M1 统计 PRD 可用率 |
| 知识库反馈能让选题越来越准 | 前 5 个小程序回顾 |
| 一个人能同时管 3 条带 | M2 后持续观察 |

## 附录：V4 → V5 核心变更

| 变更 | V4 | V5 | 原因 |
|------|----|----|------|
| 设计系统 | 列了具体 tokens 值、11 个组件名、6 种布局 | 只定义规范必须满足的要求，具体内容在 M0 中确立 | M0 前的细节全是猜测 |
| 规范定位 | "四样东西"但关系不清 | 明确四者一致性原则 + 以模板项目实际能力为准 | 防止 PRD/Rules/模板/Check 各说各话 |
| 知识积累 | 几乎没有 | 独立章节，定义积累什么/怎么用/什么节奏 | 批量试错的复利来自经验积累 |

| 冗余 | 质检项重复 3 次 | check 脚本定义一次，上线标准引用它 | 同一概念只定义一处 |
| 篇幅 | 长 | 短 ~30% | 砍掉臆想细节，保留需求定义 |
